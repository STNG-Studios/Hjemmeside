## Hvordan animationen virker — trin for trin

### 1. JavaScript-triggeren (main.js, linje 146-155)

Når brugeren klikker på et spørgsmål, kører denne jQuery-kode:

```js
$item.toggleClass('active');
```

Det er **alt** JS gør — den tilføjer eller fjerner CSS-klassen `.active` på den `<article class="faq-item">` der blev klikket på. Selve animationen sker 100% i CSS via transitions.

---

### 2. Svar-animationen (CSS, linje 592-602)

**Lukket tilstand** (`style.css:592-597`):
```css
.faq-answer {
  max-height: 0;          /* Elementet har ingen højde → usynligt */
  overflow: hidden;        /* Alt indhold der stikker ud klippes væk */
  transition: max-height 0.4s ease, padding 0.4s ease;
  padding: 0 2rem;         /* Ingen top/bund-padding */
}
```

**Åben tilstand** (`style.css:599-602`):
```css
.faq-item.active .faq-answer {
  max-height: 500px;       /* Elementet får plads til at vokse */
  padding: 0 2rem 1.5rem;  /* Bund-padding tilføjes */
}
```

**Sådan virker det:** Når `.active` tilføjes, går `max-height` fra `0` → `500px`. CSS `transition` animerer denne ændring over `0.4s` (400 millisekunder) med `ease`-timing. Samtidig animeres `padding` fra `0` bund → `1.5rem` bund over samme tid.

`500px` er ikke den faktiske højde — det er bare en "loftværdi" der er stor nok til at rumme ethvert svar. Det faktiske indhold fylder måske kun 80-150px, men `max-height` sikrer at der er plads.

**Vigtigt at vide:** `max-height`-tricket har en bivirkning — animationshastigheden *føles* ujævn. Browseren animerer fra 0 til 500px over 0.4s, men indholdet er måske kun 100px højt. Så de første 100px (åbningen) sker hurtigt, og resten af de 400px "animerer" usynligt. Omvendt ved lukning: de første 400px er usynlige, og selve lukningen sker i den sidste brøkdel af tiden, hvilket kan virke brat.

---

### 3. Chevron-animationen (CSS, linje 576-590)

**Normal tilstand** (`style.css:576-582`):
```css
.faq-chevron {
  transition: transform 0.3s ease, color 0.3s ease;
}
```

**Åben tilstand** (`style.css:588-590`):
```css
.faq-item.active .faq-chevron {
  transform: rotate(180deg);   /* Chevron drejes 180° → peger opad */
}
```

Chevron roterer over `0.3s` (300 ms) — lidt hurtigere end svaret (0.4s).

---

### 4. Sådan justerer du hastighed

| Hvad | Fil | Linje | Nuværende | Ændring |
|------|-----|-------|-----------|---------|
| **Svar åbner/lukker** | style.css | 595 | `max-height 0.4s ease` | Ændr `0.4s` til fx `0.6s` (langsommere) eller `0.2s` (hurtigere) |
| **Padding-animation** | style.css | 595 | `padding 0.4s ease` | Bør matche max-height-værdien |
| **Chevron rotation** | style.css | 581 | `transform 0.3s ease` | Ændr `0.3s` til fx `0.5s` (langsommere) |
| **Easing-kurve** | style.css | 595, 581 | `ease` | Kan ændres til `ease-in-out`, `ease-out`, `cubic-bezier(...)` osv. |
| **Max-height loft** | style.css | 600 | `500px` | Sænk til fx `200px` hvis svar er korte (giver hurtigere *opfattet* animation) |

**Easing-forklaring:**
- `ease` — starter hurtigt, bremser op til sidst
- `ease-in-out` — starter langsomt, accelererer, bremser igen
- `ease-out` — starter hurtigt, bremser langsomt op
- `cubic-bezier(0.25, 0.8, 0.25, 1)` — fuld kontrol over kurven

Fundet det. Her er årsagsanalysen:

**Problemet: `padding-bottom` springer øjeblikkeligt ved `.active`**

Sekvensen ved åbning er:

1. `$item.addClass('active')` → CSS ændrer `padding-bottom` fra `0` til `1.5rem` (24px) **øjeblikkeligt** (ingen transition på padding)
2. Men `height` er stadig `0` på dette tidspunkt
3. Fordi projektet bruger `box-sizing: border-box` (linje 24), tæller `height: 0` den *totale* boks inklusiv padding. Men padding kan ikke komprimeres til negativ — så den reelle renderede højde springer fra 0 til **~24px på ét frame**
4. Først derefter sættes `height` til `scrollHeight` og animationen starter fra ~24px

Det 24px "hop" i starten er hakket du ser — boksen vokser pludselig 24px inden den glatte animation begynder.

**Løsningen**: Flyt `padding-bottom` væk fra `.faq-answer` og over på `<p>`-elementet som `padding-bottom`. Dermed ændres `.faq-answer`'s padding aldrig, og `scrollHeight` inkluderer naturligt `<p>`'s padding.
